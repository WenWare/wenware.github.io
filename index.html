Skynet 客户端通信

概述

游戏服务器中，处理客户端的 socket 通信是核心功能之一。那么 skynet 是如何监听并连接客户端的 socket 的？连接了 socket，又是如何处理客户端消息的，如何知道需要将这个消息发送到哪个服务的呢？

Gated 服务

在启动 gated 服务的时候，里面会通过 socketdriver.listen(address, port) 监听指定的地址和端口。因为所有关于 socket 的操作都在 socket 线程中进行的，所以这里的监听需要通过管道通知 socket 线程来处理创建一个监听 socket。

socket_server_poll() 先检查是否有管道消息（通过读管道），如果有则会根据管道信息的 'L' 字段，创建一个新的 socket，并构造 epoll_event 的用户信息，里面保存着监听这个 fd 的服务信息等，然后通过 sp_add 将用户信息与 epoll_event 中的 data.ptr 中关联起来。

到这里，监听 socket 的创建就完成了，但是还需要调用 socketdriver.start(socket) 来打开 socket 的读写功能。

Gated 服务监听流程

客户端连接流程

当客户端请求连接时，socket 线程 accept 新连接，创建一个新的 socket，将这个 socket new_fd 加入到 epoll 中时，会根据老的监听 socket 中的用户信息与新的 socket_fd 关联起来。这样当客户端有数据请求的时候，socket 中的用户信息里就包含了 gated 的服务 id 了，那么就可以将数据发送到 gated 服务的消息队列中。

Skynet 服务启动机制

概述

Skynet 服务作为核心组件，主要负责处理客户端请求、服务间通信等业务功能。

在 Skynet 中，总共分两种服务，一个是 C 服务，一个是 Lua 服务，每个服务都有自己的服务 skynet_context。skynet_context 是 Skynet 中用来维护服务桩体的重要结构体，所有关于服务的操作都是通过 skynet_context 来完成的，例如查找指定服务都是通过里面的 handle 来找到对应的 skynet_context 的。

那么这两种服务都是怎么启动的呢？

C 服务启动机制

C 服务一般在编译成 .so 文件，然后在 Skynet 启动时加载或者在运行时动态加载。

动态库格式要求

这种 C 服务都有固定的文件命名以及函数命名格式（例如 service_logger.so、service_snlua.so 动态库命名），以及库中必须实现以下四个函数：

  • xx_create
  • xx_init
  • xx_release
  • xx_signal

以便在使用 dlopen 进行加载的时候能够正确找到对应的函数。

启动流程

启动或者运行时动态加载时会触发 skynet_context_new(const char * name, const char *param) —— skynet_server.c，在里面会调用到 skynet_module_query(name) —— skynet_module.c。

该函数首先会查询模块是否已经加载过，如果没有加载则会通过 dlopen 实现运行时加载动态库获取到模块对应的动态库的句柄，最终根据句柄调用 dlsym 获取到对应的函数指针，最终实现 C 服务的启动。

在获得动态库的模块后，会调用 skynet_handle_register(struct skynet_context *ctx) 进行 handle 的注册获得一个服务的专属 ID，后续根据这个 ID 进行通信，最后会对这个 skynet_context 对象进行必要的初始化（例如消息队列生成、模块的初始化逻辑）。

C 服务启动调用链

Lua 服务启动机制

Lua 服务一般在启动的时候执行 main.lua 文件时，执行 skynet.newservice(name, param) 来启动 Lua 服务。

启动步骤

  1. 服务创建请求：在执行 skynet.newservice(name, param) 最终会在 skynet.launch(...) —— manager.lua 中调用到 lua-skynet.c 里的 command 函数，然后会调用到 skynet_server.c 下的 skynet_command 执行 cmd_launch 函数，这里会执行 skynet_context_new("snlua", "clusterd") 进行 C 服务类型的 snlua 服务的启动。

  2. Snlua 服务初始化：在 snlua 服务启动完毕后，skynet_module_instance_init(mod, inst, ctx, param) 进行 Lua 服务的初始化。在里面调用到 snlua_init 函数，这里会将 launch_cb 当做第一次回调的 cb 函数，snlua_init 结束后会向 lua 服务发送第一条信息，这时已经算启动完成了。

  3. 首次消息处理：在 worker 线程中，执行服务的第一个消息时，会分发到 launch_cb 函数，这里先通过 skynet_callback(context, NULL, NULL) 将服务的 cb 设置为 null，避免重复处理，然后会在 init_cb 中执行 lua_State 的初始化，到这里，该 lua 服务是没有 callback 函数的。

  4. Lua 文件加载：在 init_cb 中，会通过 newservice 的 name 参数，使用 lualoader 的 lua 加载器进行对应 lua 文件的加载，最终使用 lua_pcall 来执行该 lua 文件。

  5. 回调函数注册：在 lua 文件中，会执行到 lua 代码 skynet.start(function() end)，最终执行到 C 代码 callback —— lua-skynet.c，这里会将 skynet.dispatch_message 这个 lua 函数注册到 registry[_cb] 中，在普通模式下，会调用 skynet_callback(context, gL, _cb) 将 lua-skynet.c 下的 _cb 函数设置为该 lua 服务的回调函数。

  6. 消息处理：当有新的消息到达时，在 worker 线程中会调用 ctx->cb(ctx, ctx->cb_ud, type, msg->session, msg->source, msg->data, sz) 从而触发到 lua-skynet.c 中的 _cb 函数，最终调用到 skynet.dispatch_message 进行业务处理。

Lua 服务启动调用链

总结

至此，Skynet 的服务启动完成。

Skynet 游戏服务器框架主体

概述

首先，Skynet 总共有 4 种线程在工作：

  1. worker 线程：消耗消息事件
  2. socket 线程：负责 socket 连接的建立与维护，以及接收与发送数据
  3. timer 线程：负责定时任务的调度
  4. monitor 线程：负责监控其他线程的状态，以及处理异常情况（如业务服务出现超时执行，例如循环 call 时，每 5 秒检测一次判断是否异常，5 秒钟内业务代码没有处理结束时，就认为是异常，这时候可以进行一些自己的输出处理，例如可以发送信号至对应的服务中打印堆栈信息）

其中，worker 线程会有多个线程，这个线程数量可以根据服务器的 CPU 核心数量才决定如何配置，一般情况下，多少个 CPU 核心数量就配置多少个 worker 线程。

Actor 模型与消息队列

Skynet 下的服务设计是基于 actor 模型的，每个服务都是一个 actor，actor 之间通过消息传递来通信，所以 Skynet 设计中会有一个全局消息队列，这个消息队列保存了所有有消息的服务的二级消息队列，二级消息队列保存的是服务的事件消息。

统一事件源

可以说所有的业务逻辑触发都是通过操作消息队列来触发的，所以 Skynet 使用了统一事件源的设计。

消息来源

  1. socket 线程：接收到客户端的消息，会根据 socket 的 fd 来判断是哪个服务的事件，然后封装好消息数据加入到对应服务的二级消息队列中，二级消息队列也会加入到全局消息队列中
  2. timer 线程：会每 2500 微秒检查一次自己的时间轮中是否有定时任务需要执行，如果有，则会根据所保存的 event 数据中的 handle 获取到对应的服务，然后将定时任务作为消息加入到二级消息队列中
  3. 服务间通信：如果是本地服务之间的通信，则会根据目标服务的服务 id 找到对应的服务 context，然后将消息加入到目标服务的二级消息队列中
  4. 系统信号：例如 kill -2 的时候需要正确关闭服务器（如通知所有玩家下线，服务正常关闭）

集群部署

另外，目前的服务器框架都应该支持集群部署，Skynet 也不例外，Skynet 支持两种集群部署：

1. Master/Slave 模式

集群中配置一台 master 服务器，所有的 slave 服务器都连接到 master 服务器上，因单点风险过大，所以目前游戏服务器基本已经废弃了。

2. Cluster 模式（推荐）

去中心化，集群中，只有在需要的时候才会主动连接到目标服务器创建维护一个 socket 连接，所有的服务器最终都会维护 n-1 个 socket 连接。

但是 Skynet 只提供了最简单的组件，Cluster 的搭建需要自己去实现：

  • 目前需要提供一个第三方来知道所有的在线服务器，这边目前可以使用 Zookeeper 来解决这个问题
  • 所有的服务器启动的时候，根据自己的配置文件，封装好自己的服务器信息，然后注册到 Zookeeper 中
  • 在需要用到 Cluster 功能时，需要从 Zookeeper（可以用 Skynet sharetable 来实现缓存）中获取到所有的在线服务器的信息，然后根据服务器的信息，来创建 socket 连接

运营相关功能

一个完整的游戏服务器，除了集群部署，还需要一些运营相关的功能，例如玩家流水，充值流水等。所以还需要部署 Kafka 来实现解耦（Kafka 不止流水功能，还有消息队列事件分发功能）去实现玩家流水上报功能。

性能监测

另外，最重要的是性能火焰图了，这里可以通过 Grafana 与 Skynet debugconsole 提供的性能输出来实现监测。

负载均衡

当然，如果有需要的话，还需要 LVS 等来实现负载均衡。

Lua 分代 GC

引言

在理解了 Lua 的增量 GC 后，接下来对分代 GC 的学习就会简单很多了。

分代 GC 的设计思路

分代 GC 的设计思路是，经过多次 minor GC 后仍然存活的对象，大概率是重要的对象，所以这些老年代对象不应该被回收，减少对这些对象的遍历，从而提高 GC 效率。

GC 的触发机制

那么，什么时候触发 youngcollection 呢？什么时候从 minor GC 切换到 major GC 呢？

与分代 GC 一样，通过 debt 与存活对象作为阈值来判断。

初始化阶段

因为触发一次 minor GC 是根据 debt 来判断的，而初始的 debt 是 0，可是分代 GC 中的 debt 设置是通过 major GC 后存活内存来计算的，所以启动 LVM 的时候，需要通过 lua_gc(L, LUA_GCGEN, 0, 0) 来触发一次 major GC。

Major GC 完成后

当 major GC 完成后，会根据存活对象的内存大小，以及 MINORMUL 参数（默认值：100（百分比））来计算下一次触发 minor GC 的 debt 阈值。

要知道的是，第一次的 major GC 是会停止执行用户代码直到 major 完成的。

Minor GC 的执行过程

当触发一次 minor GC 时，会调用到 youngcollection 函数来执行，在里面会遍历所有的对象并标记，这里有个巧妙的设计：因为新 new 的对象都是插到 allgc 的头部的，所以 allgc 链表中的对象最终会呈现出以对象年龄为顺序的。

所以在 minor GC 的时候，只需要遍历一次 allgc 就可以执行完所有的对象，并且所有的传播操作都在 atomic 中进行的，所以执行完 atomic 后，所有的对象都会被遍历完。

Sweepgen 阶段

最终，sweepgen 的时候，只需要根据上一次 GC 后的 allgc 节点与下一个年龄的起始节点来遍历，就可以回收掉所有的死亡对象，并且根据不同的年龄进行升龄操作。

带有 __GC 元方法的对象

与普通的对象一样，带有 __GC 元方法的对象，也跟 allgc 的操作一致，最终执行完一轮 minor GC。

Minor GC 到 Major GC 的切换

在执行完一轮 minor GC 后，需要根据本次的对象晋升到 OLD1 的内存总量与上一次 major GC 后的存活内存量进行判断是否需要切换至 major GC。

Major GC 的执行

如果切换到 major GC 后，其实是切换到了增量 GC 的处理中，只不过跟普通增量 GC 不同，在执行完一次完整的增量 GC 后，会判断本轮 major GC 中，被回收内存的大小与本轮新增内存量（包括待回收的内存）的百分比进行比较判断是否需要切换至 minor GC，在切换时，根据当前的存活内存大小设置 debt。

注意事项

在分代 GC 中，需要注意的是，每一次的 minor GC 都是停止执行用户代码的。

总结

至此，Lua 的 GC 学习就结束了。

通过深入分析 Lua 源码，我们可以清晰地看到 Lua 分代 GC 的完整机制：

  1. 分代设计：将对象分为新生代和老年代，减少对老年代对象的遍历
  2. Minor GC：快速回收新生代对象，停止用户代码执行
  3. Major GC：使用增量 GC 回收所有对象，避免长时间停顿
  4. 自动切换：根据晋升内存量和存活内存量自动切换 GC 模式
  5. 债务机制：通过 debt 控制触发时机，MINORMUL 参数控制阈值

这种设计使得 Lua 的 GC 既保证了内存的有效回收，又通过分代策略提高了 GC 效率，适合在服务器环境中使用。

Lua 增量 GC

引言

与所有脚本语言一样，Lua 有自己的 GC 逻辑。Lua 经历了双色标记 → 三色标记 → 四色标记（双白）→ 分代 GC 的进化过程。

目前主要使用的是三色标记与分代 GC，理解分代 GC，首先需要理解三色标记，因为分代是在增量 GC 的基础上实现的。

GC 的五个阶段

在 Lua 的三色标记的增量 GC 中，主要有以下五个阶段：

  1. GCSpause（暂停 GC）
  2. GCSpropagate（传播阶段）
  3. GCSenteratomic（原子阶段）
  4. GCSswpallgc（扫描普通对象）
  5. GCSswpfinobj（扫描有 finalizer 的对象）

其中，在原子阶段中会停止所有的世界动作的，所以在服务器开发中 GC 优化中，主要关注的就是原子阶段中所涉及到的遍历对象。

GC 的传播阶段

在 Lua 的 GC 过程中，在传播阶段中，首先会将主线程、注册表，以及所用到的元表进行标记插入到灰色链表中，然后分多次遍历灰色链表，并且根据获得的灰色对象 A 类型，遍历其引用的对象，进而将这些对象插入到灰色链表中，最终将 A 标记为黑色，不进行回收。

GC 的原子阶段

在执行完传播阶段后，会进入到原子阶段。在原子阶段中，需要对在传播阶段中被程序所修改导致进入 grayagain 链表的对象进行遍历，以及弱表的标记，与拥有 __GC 元方法的对象进行遍历。

GC 的回收阶段

执行完所有的传播阶段后，会进行对不同对象列表的遍历回收。

GC 的触发机制

上面是整个增量 GC 的过程，那么在 GC 过程中，怎么去控制什么时候启动一次完整的 GC，一次完整的 GC 中怎么分多步去进行扫描，一次扫描多少个对象呢？

首先需要关注的是：

  1. 开始完整 GC 的触发点：是根据 debt（单位是 KB）来触发的
  2. 在 GC 过程中的每一小步：根据 stepsize（单位是 KB）来进行触发
  3. 在每一小步 GC 中：stepmul 决定了一小步需要处理多少数据（单位是 个 —— 这里的个是根据不同阶段的意义不同）
  4. stepsize 的计算：默认值是通过 200 个 table 大小来计算的，最终约等于 17，之后用于计算处理对象数量得到 work2do = 2，这里的 200、17、2 是通过经验调优所得
  5. 一次 singlestep 的功能：是遍历 work2do 个对象的所有引用，所以这里执行后遍历的内存大小是可能远大于 17KB 的，所以不用担心一次步骤进行的内存扫描追不上期间产生的内存导致内存泄漏

GC 的债务机制

那么总结上面的，第一次 GC 是在第一次申请内存的时候执行的，因为初始的 debt 是 0。开启了一次完整的 GC 后，在默认情况下，每一小步的 GC 后产生了 17KB 的内存后，会再次触发增量 GC，每次增量 GC 会处理 2 个工作单元。

当执行完一次完整的 GC 后，在默认值的情况下，会将 debt 设置为当前存活的所有对象加起来的内存大小，也就是说这次存活对象如果是 30MB，那么下一次需要累积申请了 30MB 以上的内存才会再次触发一次完整的 GC。

GC 触发流程

总结

上面就是整个的增量 GC 的过程。

通过深入分析 Lua 源码，我们可以清晰地看到 Lua 增量 GC 的完整机制：

  1. 四色标记：使用白色、灰色、黑色三种颜色标记对象状态（双白用于分代）
  2. 分代回收：基于增量 GC 实现分代回收
  3. 五个阶段：暂停、传播、原子、扫描普通对象、扫描 finalizer 对象
  4. 债务机制：通过 debt 控制完整 GC 的触发时机
  5. 增量执行：通过 stepsize 和 stepmul 控制每次 GC 的工作量

这种设计使得 Lua 的 GC 既保证了内存的有效回收，又避免了长时间的停顿，适合在服务器环境中使用。

  2. lua_loadfile 加载完脚本后，里面的数据是如何在 lua_pcall 里执行的？

lua_pcall 的执行原理

执行流程概述

从 lua_loadfile 中获得脚本语言的闭包后，这时候可以通过 lua_pcall 执行 Lua 脚本的字节码列表了。

在一系列的调用过程中，保存当前的栈状态以及 hook 等设置后，最终会在 f_call（lapi.c）中调用到 luaD_callnoyield（ldo.c）转发调用 ccall。

CallInfo 的生成

ccall 里则调用了 luaD_precall（ldo.c），来根据编译过程中的 Proto 数据，生成一个新的 CallInfo 并初始化（这里只列出了 Lua 函数的执行）。

字节码执行

接着根据生成的 CallInfo 调用了 luaV_execute（lvm.c），来执行字节码指令。在这里，如果是函数调用，则会先保留当前的栈状态，然后根据 CallInfo 的字节码执行不同的命令解析，遇到新的函数调用的时候，也会调用到 ccall 进行执行。

栈帧管理

在这里可以看出，CallInfo 是作为 Lua 的栈帧的，CallInfo 的生成是在执行具体的函数调用字节码的时候才会去生成，并非一开始就生成了所有的 CallInfo。

整个 Lua 脚本的执行过程就是通过 CallInfo 来管理栈帧的，每个栈帧对应一个函数调用，栈帧之间通过 CallInfo 来传递参数和返回值。

最终，整个 Lua 脚本执行完毕。

完整调用流程

总结

至此，Lua 是怎么运行起来的，已经有了较清晰的脉络了。

执行一个 Lua 脚本主要分了三个阶段：

  1. 创建 Lua State：创建一个 lua_state 并初始化库，例如 “require”、”math”、”io” 等库，最重要的则是初始化 _G 表与 registry 表

  2. 加载脚本文件：使用 lua_loadfile 加载 Lua 脚本文件，执行词法分析、语法分析、语义分析，将代码编译为字节码指令，并保存到一个 closure 中返回

  3. 执行字节码：执行 lua_pcall 执行闭包中的字节码指令去操作 global_state 中的 stack，最终返回

这种设计使得 Lua 虚拟机具有良好的性能和可维护性，通过分层的方式实现了从脚本加载到字节码执行的完整流程。

lua_loadfile 的实现原理

文件读取机制

当在加载文件的时候，必须要对文件进行读取，然后一个文字一个文字进行解析。这里用到了 LoadF 与 ZIO 两个结构体进行文件的读取。

为什么要用到两个不同的结构体？

  • LoadF 的作用：主要是对源文件的读取
  • ZIO 的作用：对已加载的文件进行读取（即对 LoadF 中的缓存数据进行处理）

这样做 ZIO 就可以将不同文件系统的读取方式分离开来。这种设计在我们日常编码中有很重要的作用，将可变与不变的功能区分设计，也是读源码的知识增长点。

解析过程

实际的文件解析主要发生在 luaY_parser（lparser.c）里，里面会对 Lua 脚本文件进行词法分析并最终维护一个 FuncState 结构体，最终返回包含 FuncState 里的 Proto 数据的闭包。

这里执行了对 mainfunc 的解析，需要注意的是，这里的解析是一个词一个词进行解析的，由 luaX_next 获取当前的词，之后会对脚本文件的代码块进行 statlist 解析。

在 statlist 的解析过程中，会根据不同的词，去进行不同的语句解析：

  • 例如遇到 local 关键字，就会调用 localfunc 解析 local function 语句
  • 或执行 localstat 语句

而在具体的 localstat 的解析过程中，会针对不同语法情况，解析并维护 expdesc 结构体，用于存储局部变量的类型、索引等信息。这个结构体在最终的字节码生成中会使用到并生成最终的字节码指令保存到 FuncState 的 Proto 数据中。

完整调用流程

总结

通过深入分析 Lua 源码，我们可以清晰地看到 Lua 脚本的加载和执行流程：

  1. 文件读取：使用 LoadF 和 ZIO 结构体实现文件读取的抽象
  2. 词法分析：luaX_next 逐词读取脚本内容
  3. 语法分析：luaY_parser 解析语法结构
  4. 字节码生成：生成 Proto 数据结构
  5. 闭包创建：luaF_newLclosure 创建 LClosure
  6. 闭包入栈：将生成的闭包压入 Lua 栈

这种分层设计使得 Lua 的虚拟机具有良好的可扩展性和可维护性。

Lua中的require执行流程

引言

一直以来，对 Lua 的源码还是没有太深究下去，只是研究了中等，知道是怎么执行的，但是都没有具体的去看，只是停留在了知道是怎么执行的这个阶段。现在的 AI 写代码已经是必然趋势，如果只是知道其设计原理，感觉还是不太够的。

所以，最近在研究 Lua 源码武装自己，希望能在未来的项目中，能够更加游刃有余的处理 Lua 的代码。

Lua 源码已经看了大概，加上 AI 的帮助，确实比之前的自己硬啃要快很多，也更有目的性。现在先对 Lua 的 require 的实现逻辑进行总结。

luaL_openlibs 的作用

首先，要理解 require 是怎么执行的，必须先要知道在启动一个 LVM 时的 luaL_openlibs 做了哪些东西：

  1. 注册了一些全局函数，比如 print、require 等
  2. 加载了一些基础的库，比如 math、string、table 等
  3. 加载了一些可选的库，比如 socket、lfs 等
  4. 加载了一些自定义的库，比如 agent 等
  5. 初始化了一些全局变量，比如 _G、package 等

在 luaL_openlibs（linit.c）中，会对 luaL_Reg stdlibs[] 里的库进行加载，而其中的 {LUA_LOADLIBNAME, luaopen_package} 项，会加载 package 库。

package 库中会注册 require 函数，所以，require 函数就被注册到了全局环境中，执行完后，require 就已经保存到 _G 中。

require 的执行流程

当 LVM 执行 Lua 脚本时，遇到 require 语句时，会先从 package.loaded 中查找是否已经加载过该库，如果没有加载过，就会调用 require 函数加载该库。

这时候执行到 require 命令的字节码时，会调用到 ll_require 函数（loadlib.c），会先从 package.searchers 中查找到与之对应的加载器，例如：

  • Lua 文件的加载器对应 LUA_LOADLIBNAME
  • C 库的加载器对应 LUA_COLIBNAME

找到加载器后，就会调用加载器加载该库：

  • Lua 文件的加载器会调用 luaL_loadfile 加载该文件
  • C 库的加载器会调用 luaopen_xxx 加载该库

加载完成后，会将该库保存到 package.loaded 中，以便后续的 require 调用可以直接从 package.loaded 中查找。

最终，会调用到库中的 luaopen_xx 函数，里面会创建一个新的 table，这个 table 里会有对外接口的函数映射，require 返回对应的库的 table，例如 require("math") 会返回 math 库的 table。

完整调用流程

总结

通过深入研究 Lua 源码，我们可以清晰地看到 require 的完整执行流程：

  1. 初始化阶段：luaL_openlibs 加载 package 库并注册 require 函数
  2. 缓存检查：首先检查 package.loaded 是否已加载该模块
  3. 加载器查找：遍历 package.searchers 找到合适的加载器
  4. 模块加载：根据模块类型（Lua 或 C）调用对应的加载器
  5. 结果缓存：将加载结果保存到 package.loaded 中

这种设计既保证了模块只加载一次，又支持多种加载方式，是 Lua 模块系统的核心机制。
