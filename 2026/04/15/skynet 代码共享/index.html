Skynet 代码共享

概述

之前在讲 Lua 加载到执行中提到过，Lua 的加载过程是通过语法分析、词法分析、语义分析后，生成了一个包含 Proto 的 closure 保存到 luaState 中，然后在运行时，通过 Proto 来遍历执行字节码并生成对应的栈帧 callinfo 来对 luaState 里的 stack 进行操作。

那么 Skynet 中，一般会有多个 agent 来运行玩家的代码逻辑，每个 agent 都需要运行到所有的相同的代码，所以 Skynet 在加载代码时，对原生的 Lua 代码进行了改进，添加了一个共享代码的 CC lua_state 来保存所有文件对应的 function Proto。

如代码所示，如果在编译 Skynet 时设置了 LUA_CACHELIB 定义，那么在 skynet_main.c 中会初始化一个对 CC 的自旋锁，后续用来对 CC 进行读写操作。

Codecache 服务

另外，在创建一个 Lua 服务时，根据 LUA_CACHELIB 定义，定义后使用修改版 Lua 提供的 luaopen_cache，实现真正的 Proto 共享。

Loader 加载流程

在初始化 Lua 服务时，调用了改版的 luaL_loadfile(L, loader); 加载了 loader.lua 脚本。当执行到 lua_pcall(L, 1, 0, 1); 时，在 loader.lua 中会根据 newservice 的参数，来获取到服务文件名进行 loadfile(filename) 最终执行到 luaL_loadfilex。

Proto 缓存机制

最终，通过加载服务的 Lua 文件时，对代码使用到的 Lua 文件进行 luaL_loadfilex 时，判断是否有该文件名的 Proto 缓存。如果有则引用数据到当前 lua_state 中的 closure 中并返回，否则加载文件并将 Proto 保存到 CC lua_state 中。

lauxlib.c

总结

实际上 Skynet 通过维护一个全局的 CC 中的 lua_state 变量来实现代码共享的，在该进程中的所有 LVM 在加载 Lua 代码时，都会先从 CC 中获取已有的 Proto 并引用到当前 LVM 中，避免了重复加载和内存泄漏。

另外，Skynet 热更的时候，除了需要关注 package.loaded 表以外，额外需要关注 CC 中的 Proto 缓存，来确保所有 LVM 中的 Proto 都是最新的。
