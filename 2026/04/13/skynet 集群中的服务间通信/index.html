Skynet 集群中的服务间通信

概述

游戏服务器中，通常都必须有集群配置的，所以服务间通信，除了需要处理单节点的情况外，还需要考虑集群的情况。在 Skynet 中，如何处理集群中的服务间通信？

在开始前，需要对 cluster.lua、clusterd.lua、clustersender.lua、clusteragent.lua、clusterproxy.lua 这 5 个文件的功能进行了解。

Cluster 架构图

可以看到，clusterd.lua 是协调者，负责协调服务之间的通信。因为不同节点的服务，通信最终都会通过 socket 来实现，那么，最终的通信管理都离不开 gate 的维护。

Cluster 服务启动

游戏服务器启动的时候，会启动一个 clusterd 服务，负责协调服务之间的通信。该服务会启动一个 gate 服务，该 gate 服务会监听一个端口，用来接收不同节点的 socket 连接。gate 服务在启动的时候会通过 handler.open(source, conf) 来初始化并用闭包 watchdog 缓存 clusterd 服务的服务 id，后续有连接到该端口时，会通过 watchdog 来转发到 clusterd 服务中，从而用 clusterd 来维护连接。

处理流程

1.1 当获取 local s = sender[node] 时，cluster 会通过元表向 clusterd 获取目标节点对应的 clustersender 服务，如果没有则创建一个 clustersender 服务并连接到服务 B 的远程节点。

1.2 服务 B 的 gate 服务在收到连接请求时，gateserver.lua 文件会处理 MSG.open 事件，从而在 gate.lua 中调用到 skynet.send(watchdog, "lua", "socket", "open", fd, addr) 转发到 clusterd 服务中。clusterd 服务收到并处理 open 逻辑，会创建一个 clusteragent 服务（clusteragent 服务在启动时，会通过 skynet.call(gate, "lua", "forward", fd) 来将该 socket 连接的 fd 绑定到 gate 服务中，后续的 socket 数据都会通过 gate 服务转发到对应的 clusteragent 中）。目前为止，cluster 的 socket 连接完成了。

1.3 服务 A cluster.send 给服务 B 发送数据，实际是 A 节点中的 clustersender 服务收到 push 命令后，通过 socketchannel 的 channel:request() 将数据写入缓冲区中等待 socket 发送。由于 response 参数为 nil，不会等待响应。

cluster.call

处理流程

2.1 当获取 local s = sender[node] 时，cluster 会通过元表向 clusterd 获取目标节点对应的 clustersender 服务，如果没有则创建一个 clustersender 服务并连接到服务 B 的远程节点。

2.2 服务 B 的 gate 服务在收到连接请求时，gateserver.lua 文件会处理 MSG.open 事件，从而在 gate.lua 中调用到 skynet.send(watchdog, "lua", "socket", "open", fd, addr) 转发到 clusterd 服务中。clusterd 服务收到并处理 open 逻辑，会创建一个 clusteragent 服务（clusteragent 服务在启动时，会通过 skynet.call(gate, "lua", "forward", fd) 来将该 socket 连接的 fd 绑定到 gate 服务中，后续的 socket 数据都会通过 gate 服务转发到对应的 clusteragent 中）。目前为止，cluster 的 socket 连接完成了。

2.3 服务 A cluster.call 给服务 B 发送数据，实际是 A 节点中的 clustersender 服务的 req 通过 socketchannel 来将数据写入缓冲区中等待 socket 发送。发送完毕后会通过 response（表示是否需要等待返回数据）判断是否需要调用 wait_for_response 来等待服务 B 的处理结果。

2.4 节点 B 中的 socket 线程将请求分发到 gate 服务中，gate 服务根据 fd 来将数据转发到对应的 clusteragent 服务中，clusteragent 服务将数据转发给服务 B 进行处理。节点 B 中的请求处理完成后，clusteragent 将处理结果通过 socket 直接写到缓冲区中等待发送。

2.5 socketchannel 的 dispatch 协程在监听到 socket 有数据可读时，从缓冲区中读取响应数据，并唤醒等待中的协程，继而返回给服务 A。

cluster.send 与 cluster.call 的区别

cluster.call 中，与 cluster.send 的区别在于，在 cluster.call 中，最后的逻辑是走的 skynet.call。那么在 clustersender wait_for_response 返回后，会将处理结果返回给服务 A，这里的返回，则走 skynet.call 的返回机制。

注意事项

到这里，cluster 的通信结束了。这里需要注意的是，clustersender 服务中，socketchannel 的接收数据是不经过 epoll 的（由独立协程阻塞读取），而发送数据则是通过 socket.write() 直接写入缓冲区。
